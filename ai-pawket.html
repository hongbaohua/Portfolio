Pawket（喵喵財庫）是一個以貓咪為主題的財務管理 Web App，從無到有全程以 Prompt Engineering 驅動開發。整個過程分為三個階段：先以「30年投資理財專家」角色引導 Gemini 透過 16 輪問答完成功能架構決策，再以「20年 UI/UX 設計師」角色制定貓咪主題視覺規範，最後將完整規格 Prompt 交付 Google AI Studio 生成程式碼。App 整合 Gemini API 實現帳單 OCR 辨識，以吉祥物 Meowney 作為財務健康情緒指標。

Prompt 工程設計策略

角色設定：讓 AI 成為領域專家

這個專案使用了兩組核心角色設定，分別負責功能架構與視覺設計兩個維度。角色扮演的關鍵在於讓 AI 從「工具」轉化為「顧問」，使其在回答每個問題時都帶入專業立場與設計邏輯。 
【功能架構角色設定】
你是一位年資30年專業的投資理財專家，正在著手準備製作一個
「掃描對帳單後，並自動分類費用項目」的 app
現在正在 app 的研發初期，請用一次只問一個問題的方式，逐漸擴充完成這個 app 的功能

【UI/UX 角色設定】
你是20年經驗的網頁平面設計師，正在為這個專業財務 App 設計 UI/UX
請以客群為年輕女性為主，打造輕巧、活潑、可愛的元素氛圍
準確提供給 AI Studio 的 prompt

【命名決策】
重新命名財富貓咪，使其更有創意性（財富、貓咪基礎元素不變）
→ Gemini 提出 Purrse / Meowney / Pawket
→ 選定 Pawket（Paw + Pocket）、吉祥物命名 Meowney（Meow + Money）
「一次只問一個問題」的問答推進法

刻意在 Prompt 中限制 Gemini 的回應方式——每輪只問一個決策問題，強迫對話以遞進的邏輯展開。這讓每個功能決策都有清楚的原因，而非一次丟出大量模糊需求。共進行了 16 輪問答，涵蓋從資料模型到警示公式的完整架構設計。

16 個功能架構決策

# 決策問題 選擇方向 設計邏輯
1鎖定哪類對帳單信用卡＋簽帳金融卡切入消費核心，黏著度高
2分類優先策略大類別歸納（Category First）讓用戶 3 秒看到圓餅圖，建立信任感
3模糊商家處理預設最大機率 + 允許手動拆單流暢度優先，保留專家介入能力
4分類體系現金流三大指標（固定/變動/投資）直接服務預算編列目標
5子分類深度詳細三級分類「不是餐飲花太多，是咖啡花太多」的精準洞察
6拆單方式手動快速填寫（不要求收據）降低操作摩擦力
7分析定位主動預算監控 + 警告財務教練（事前干預），而非帳本（事後報告）
8預警機制動態時間加權警示依月份已過天數比對消費進度，更早預警
9OCR 驗證方式針對性高風險驗證（信心度 < 85% 才標記）效率優先，不強制全欄位確認
10預算初始值智能歷史建議數據驅動，起始預算就精準
11主儀表板焦點變動支出警示聚焦立即行動導向，看到就知道要處理什麼
12初始預算設定智能歷史建議預算與實際生活吻合
13數據儲存設備端優先 + 可選備份下載最高隱私承諾
14導出格式PDF 優先 + CSV 次要PDF 視覺衝擊、CSV 專業分析
15MVP 必備導出PDF 報表優先視覺衝擊力利於分享宣傳
—追加修正新增收入辨識（Income 分類體系）簽帳金融卡有轉入/存款等收入，需區分收支

核心公式：動態時間加權警示

由 Gemini 理財專家角色設計，這個公式解決了傳統記帳 App「月底才知道超支」的問題。以當月已過天數為基準，即時推算消費速率是否超出預算節奏。 
當前已花費 / 當月已過天數  >  當月總預算 / 當月總天數 × K（K = 1.15）

超標率 > 20%  → 🔴 Critical（紅色警示）
超標率 ≤ 20%  → 🟡 Warning（黃色警示）

例：月預算 $12,000，已過 15 天，已花 $8,000
當前日均：8,000 / 15 = $533
預算日均上限：12,000 / 30 × 1.15 = $460
超標率：(533 - 460) / 460 = 15.9%  → Warning

App 三頁面架構

① 貓咪指揮中心（Dashboard）

核心儀表板，以 Meowney 吉祥物情緒指標（Smile / Meh / Frown）呈現當前財務健康狀態。包含動態時間加權警示、DTI 償債比率、財務結構圓餅圖（固定/變動/投資）、潛在財富機會試算（變動支出投入 7% 年化報酬 10 年後獲利估算）、夢想目標進度條（4 段動畫：種子→成長→衝刺→豐收）、異常消費偵測、季節性支出熱圖，以及 PDF 報告匯出。
② 餵食帳單 Scanner

上傳銀行或信用卡帳單截圖，Gemini API 自動執行 OCR 辨識，智能區分收入與支出，標記三級分類。Prompt 設計採「主表優先原則」避免摘要欄位重複計算，並加入指數退避重試機制（最多 3 次，1s → 2s → 4s）。
③ 罐罐明細本（交易明細）

全文搜尋、L2+L3 標籤篩選、拆帳（分裝）功能（含 Undo 5 秒復原）、批次修正相似交易、JSON 匯入/匯出備份、手動新增交易。三級分類體系（L1 → L2 → L3）讓支出可以細緻追蹤到「咖啡」而非只有「餐飲」層級。

開發挑戰：AI Studio 的無法復原限制

Google AI Studio 的對話是覆蓋式的——每次修改都直接覆蓋前一次的程式碼，沒有 Undo。這導致「修好一個功能，後續對話又把其他功能改壞」的情況反覆發生。應對策略是在每次重大修改前手動另存一個新版本資料夾，最終累積了 9 個版本。
版本主要更新
v1.1初始雛形，基本三頁框架
v2.1UI/UX 重構，導入貓咪主題視覺
v3.2新增夢想目標功能、標籤系統
v3.3統一頁面命名、新增快速查找
v4.2首頁分析邏輯修正
v4.3目前最新版本，多項 Bug 修正；Claude Code 接手後持續優化
版本演進中最難解決的問題是夢想目標進度條邏輯錯誤：原本的加權公式（70% 存款進度 + 30% 時間進度）導致「還沒存任何錢但快到期時，進度條自動顯示 24%」的誤導性結果。此問題在 Claude Code 接手後，透過直接閱讀 logicService.ts 找到根因，將 weightedPercent 改為直接等於 financialPercent（實際存款比例）才得以修正。

過程反思

這個專案最大的收穫是驗證了「Prompt Engineering 作為產品設計方法」的可行性——透過角色設定和問答式推進，讓 AI 扮演理財專家完成了一份紮實的功能規格書，而不是讓設計師憑空想像功能需求。16 個決策點的問答過程，其實就是一份結構化的產品思維文件。

AI Studio 的無法復原限制則是一個重要的課題：它讓我學到「AI 生成工具的使用策略」和「版本管理的必要性」。每次修改前先備份、每個 Prompt 都要明確指定範圍，是降低風險的關鍵。

最後接手 Claude Code 修正 Bug 的過程，讓我體驗了不同 AI 工具在同一個專案中各自扮演的角色——Gemini 適合發散思考與規格設計，AI Studio 適合初始程式碼生成，Claude Code 適合精確的除錯與邏輯修正。三者組合，才是完整的 AI 輔助開發鏈。
