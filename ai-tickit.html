Tickit（通往及格的通行證）是一個為備考「品牌企劃師初級」自製的線上刷題 Web App，後擴充至 ERP 軟體應用師。先以 Prompt Engineering 引導 Gemini 以嚴格 Markdown 格式生成題目與解析，再以完整架構規劃驅動 Claude Code 開發純前端應用。平台支援多使用者識別、主題與單元混搭出題、隨機抽題（含題組保護邏輯），完成後公布成績並自動分析弱點，所有紀錄永久保存於 Supabase 雲端資料庫。開發過程從 Python/Streamlit 全面重構為純前端 HTML/CSS/JS + GitHub Pages，並歷經 parser bug 修正、版面置中重構、UI 品牌化等多輪迭代。

我在這個專案中做了什麼

規劃與驅動，而非執行程式碼。

這個專案中，我負責的是整體架構設計與 Prompt 策略——設計 Prompt 框架讓 Gemini 扮演出題者、以完整開場白向 Claude Code 說明架構需求、主導每次版本迭代方向（重構技術棧、UI 品牌化、部署遷移）、以及命名品牌 Tickit（Tick + it + Ticket，ck 連筆形成勾號為 Logo 核心）。Supabase 雲端資料庫設定、GitHub Pages 部署、GitHub 日常維護也全數自主完成。

Prompt 策略：讓 Gemini 成為出題者

Gemini 出題的難點在於格式控制——題目、選項、答案、解析必須在同一份 Markdown 檔案中，格式嚴格符合 parser.js 的解析規則。第一次要求時格式不符（選項換行錯誤、分隔字串不一致），追加 Prompt 明確指定格式後才達標。

架構規劃如下：
首頁：用戶識別（輸入代號，無需密碼）；選擇主題與單元（支援多選混搭）；
      選擇全部或隨機 N 題。
測驗頁：逐題顯示，完成後才公布成績（答對 / 總題數）；
        列出錯題與解析；可展開所有解析。
考生管理頁：過往測驗紀錄；弱點分析（依題型統計錯誤模式，提供強化建議）。

完成後上傳 GitHub，部署至 GitHub Pages（靜態網站，docs/ 資料夾）。

三頁面功能架構

① 首頁（index.html）— 測驗設定

用戶代號識別（無需密碼，輸入代號即建立或讀取帳戶）、選擇測驗主題（自動讀取 manifest.json）、選擇單元（多選混搭，即時顯示各單元題數）、選擇題數（全部 / 隨機 N 題）。隨機模式以題組為單位抽取，不拆散「承上題」情境題。
② 測驗頁（quiz.html）

逐題顯示題目與四選項（不即時顯示對錯）、進度條 + 題號標籤。完成後公布成績（答對數 / 總題數），預設列出錯誤題目與解析，可展開查看全部解析。弱點摘要（單元答對率 < 75% 自動標記），成績自動寫入 Supabase 雲端資料庫。
③ 考生管理頁（dashboard.html）

下拉選單選擇考生、關鍵指標卡片（測驗次數、平均答對率、累計題數、最高答對率）、過往測驗紀錄列表（含回顧錯題功能）、答對率趨勢折線圖（Chart.js）、弱點診斷（各單元三色分類 + 強化建議）、各單元答對率橫條圖、高頻錯題 Top 10。

開發挑戰一：Parser Regex Bug

問題：Q2 整題從測驗中消失。

ERP 共同科目題庫中，某題的解析以數字條列格式呈現（1. R/2、2. ERP ECC），而 parser.js 解析題號的 regex 為 /^\s*(\d+)[.、\s]\s*/gm，這個格式完全符合 regex 的題號判斷條件——parser 把解析裡的子項目誤判為新題號，導致 Q2 被「切斷」，整題從測驗中消失，且 Q2 的選項變成一道沒有題目文字的孤立題。

修正方式：將解析中的數字條列改為括號格式（(1) R/2），避免觸發 regex。這次 bug 讓我深刻體會到：題庫格式的每個細節都可能影響 parser 行為，Markdown 格式規範必須納入題庫管理流程。

開發挑戰二：版面「數學正確但視覺偏左」

問題：主內容區明明置中計算，但視覺上仍偏向左側。

第一次嘗試：側欄使用 position: fixed，主內容以 padding 公式補償（padding: 2rem max(1.5rem, calc((100vw - 220px - 680px) / 2))）。數學上是對的，但因為側欄佔據了左側固定空間，「右側剩餘區域的中央」並非「整個畫面的中央」，視覺感知還是偏左。

真正的解法：讓側欄回到文件流（position: sticky），整個 .layout 容器設 max-width: 1280px; margin: 0 auto，主內容區再以 align-items: center + .main > * { max-width: 660px } 控制寬度。側欄與主內容並排，整體容器在畫面中央，才真正達到視覺置中。

過程反思

這個專案最大的收穫是親身體驗了「設計師如何用 AI 工具完成一個完整的軟體產品」——從需求定義、Prompt 策略、架構規劃，到每輪迭代的問題排查，都需要清晰的系統思維，而不只是會下指令。

Gemini 出題 + Claude Code 開發這樣的工具分工，讓我理解不同 AI 工具有各自的強項：Gemini 擅長創意發散與知識生成，Claude Code 擅長理解程式架構脈絡並精準修正。選對工具、給對 Prompt，才能讓 AI 真正發揮作用。

最後，Tickit 不只是一個備考工具——它本身就是一件 AI 協作作品，記錄了從零開始驅動 AI 開發完整軟體產品的完整思考過程。
